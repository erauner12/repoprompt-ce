# Mock Iteration Spec — RepoPrompt_Command_Center.html

Companion spec for the v2 interaction prototype. Kept in lockstep with the mock and `Mission_Demo_Scripts.md`: any change to the mock updates this spec (and the demo script) in the same pass.

**Terminology.** **Director** is the decided term, everywhere: the mode is Director (`⌘2` pill), a Director runs **Missions**, and the mock's internal vocabulary is fully renamed to match (message role `director`, `directorChild`, `directorDirect`, `directorSessions()`, `.director-shell`, `state.surface="director"`, etc.). The Swift codebase's files are still named `Coordinator*` (`CoordinatorModeView.swift`, `CoordinatorModeViewModel`) — §6/§11 cite those current names; the Coordinator→Director rename in Swift lands as part of adoption. Vocabulary: *Mission* = the Director-owned control plane, *Session* = an individual Agent run, *Delegated Session* = an Agent session launched by a Mission, *Sub-agent* = a child spawned by an Agent session (never directly by the Mission).

**Iteration delta (v3.16.4 → v3.16.5).** **THE root cause — the page could never scroll.** Desktop's `html, body { height: 100% }` was never released under the media query; combined with `overflow-x: hidden` on body, iOS caps the document scroller at exactly one viewport — so **nothing below the fold was reachable on any view, all night**. One-line fix: `html, body { height: auto; min-height: 100% }` on mobile. Every earlier flattening/releasing fix remains correct — they were necessary but masked by this ceiling. Lesson recorded: when multiple views share one impossible symptom, audit the ROOT (html/body) first, not the leaves.

## 1. Intent

This iteration proves the full Mission lifecycle as an interaction contract, not a layout: directive-first missions (the Director infers the plan's shape from the directive and announces it; user presets are trust policies, never topology), evidence-gated step completion with an accumulating Mission evidence ledger, supervised-to-a-bar execution for both writing and reading steps, real graph shape (parallel bands with independent per-thread steering and one-at-a-time convergence, bounded loops), mid-flight topology escalation (`⑂` split), reversibility-gated closing actions, and fleet-level surfaces (Decisions queue, All Agents Board) fed only by user-produced work. The mock is a runnable simulation of this contract so each mechanic can be demoed and critiqued before Swift implementation.

## 2. Major UX Changes

(Deltas from the v1 layout mock.)

- **Change (v2.3):** Directive-first start — shape is inferred, never picked. The empty Director state leads with "describe the mission"; the Director reads the directive, picks one of five internal shapes (`SHAPE_LIBRARY`), and announces it in grounding with a fixed format ("This reads as a {shape} — {reason}… Not the shape you meant? Say so below before I draft."). Pre-plan composer messages re-run inference over directive + all steering; a changed result is announced. Post-draft, shape changes are plan revisions.
  **Why:** If the Director can draft a plan it can pick the shape; a picker made users choose topology abstractions before typing a word, contradicting "the plan is an output." The inference announcement is also the stronger intelligence demo.
  **User-visible result:** Type first; the shape callout appears in grounding; the approval checkpoint names the inferred shape; the plan overview reads `**Shape:** … (inferred from your directive)`.
  **Risk/confusion:** Misinference — mitigated by the reason + explicit pre-draft override invite; the five-shape set and precedence are a closed contract (design doc §2).

- **Change (v2.3):** Templates become **Mission Policies** — how much you trust a run: pace, plan-approval/gate toggles, an *optional* Definition of Done, budget, pinned skills. Built-ins **Default** (Step) and **Hands-off** (Auto); "＋ New Policy" opens a trimmed editor (no brief, no plan preview). Empty Done defers to the drafted shape's close; non-empty Done re-targets the land node's terminal (title/desc/terminal replaced, reversibility gate follows). Skills = inferred-from-directive ∪ policy pins.
  **Why:** The only durable thing worth saving across missions is trust configuration, not topology.
  **User-visible result:** `Policy · …` composer chip; policy cards with default chips and a `closes by:` / `close: per drafted shape` chip; picking pre-loads composer toggles (which the user can still override per mission).
  **Risk/confusion:** Users wanting a saved *shape* have no artifact for it — deliberate deferral (design doc §7).

- **Change:** Evidence gate + Mission evidence ledger. Every node carries "Done when"; completion posts a green evidence card and a "Evidence meets the bar — advancing" line; the Plan view has a persistent ledger panel.
  **Why:** A step is done on evidence, not because a workflow returned; makes Request-changes precise (reject against a stated bar).
  **User-visible result:** ✓ evidence cards in the thread; `✓ node — proof` rows accumulating in the Plan.
  **Risk/confusion:** Could feel heavy for tiny steps (see §13). v2.2 note: the gate now demonstrably **rejects** (amber ✗ card + same-session re-steer on the Scoped Change implement step), so completion is no longer shown as self-graded happy path.

- **Change (v2.3):** Glyphs mark **shape variance only**: `⇉` parallel band tag, `↳` convergence tag, `⟳` loop strip. Supervision-to-a-bar is the stated universal rule (and demonstrated by the ✗ bar-miss beat), so the per-row `↻` marker is retired — it appeared on nearly every row, which made it noise, and the picker previews that carried it are gone with the picker.
  **Why:** Six glyphs of taxonomy for a community proposal was too much; three that always mean "the shape differs here" are learnable at a glance.
  **User-visible result:** Probe/loop/parallel beats still play live in the thread; plan bands carry ⇉/↳; loop nodes carry ⟳; nothing else is glyphed.
  **Risk/confusion:** None — the removed glyph encoded a property that was true everywhere.

- **Change:** Mid-flight topology escalation — `⑂ Propose split` on the plan turns a single Implement into two `partWorktree` builds + Integrate, or "Keep as one."
  **Why:** Templates bias the shape but don't cage it; the same worktree judgment as Spec-First, reached during the run.
  **User-visible result:** A split suggestion strip on the Plan; a `splitProposal` checkpoint; the plan rewrites in place and the overview gains a "Revised mid-flight" note.
  **Risk/confusion:** Presenter-triggered (a button) in the mock — mock-only trigger; a real Director would surface it from discovery evidence.

- **Change:** Reversibility-gated closing instead of an outcome enum. Free-text Definition of Done; the Director drafts the closing action ("will close by opening a PR — ⚠ merge is irreversible, you'll confirm"); PR/issue/report are examples, not types.
  **Why:** What matters for a checkpoint is reversibility, not which of three buckets the close is.
  **User-visible result:** The **policy editor**'s optional Done shows the drafted close + reversibility hint when non-empty (empty = "each mission closes the way its drafted shape ends"); merge always stops; issue/report closes don't.
  **Risk/confusion:** Users wanting to pin the exact closing action have only the policy Done text to steer it (open question #13).

- **Change:** Steer-over-spawn everywhere. Re-passes (Request changes at merge, PR feedback, step refinement) reuse the same implementation/review sessions; boards never grow duplicate cards. Tests fold into Implement's bar (produces a *tested diff*) — no separate tests step.
  **Why:** Worktree-orchestration playbook: steer beats spawn; a session is a durable thread.
  **User-visible result:** "Steering the same … session" director lines; `↳ Director steered this session` messages inside the Agent transcript; board stays clean across passes.
  **Risk/confusion:** None remaining — v2.1 unified both Request-changes paths on steer (sessions kept on every re-pass).

- **Change:** Fleet surfaces are earned: Decisions queue and All Agents Board show only user-produced work; seeded demo content lives under collapsed **Examples** in both rails.
  **Why:** Fixes the demo "cold-open churn" note; also a real product principle (your queue is yours).
  **User-visible result:** Clean start on reload; example missions/sessions never appear in Decisions/Board.
  **Risk/confusion:** None significant; example missions still show their own sessions on their own plan/board when opened.

- **Change:** Pre-plan probe steering via composer text (replaces a "Gather evidence" button). At the approval checkpoint, a message names areas to probe → cyan Probe card → folded into the draft ("Also scoped (you pointed me at): …").
  **Why:** Free-text-over-buttons, consistent with the rest of the surface.
  **User-visible result:** Probe card + director acknowledgment before drafting.
  **Risk/confusion:** Discoverability relies on the composer placeholder text.

- **Change:** Rendered plans — overview Markdown + Mermaid diagram; node `desc`/`detail` are Markdown with per-node diagrams; plan revisions extend the overview diagram.
  **Why:** The same rendered plan serves human review and agent handoff context.
  **User-visible result:** Diagrams in Plan overview and node cards; revision adds a node to the flowchart.
  **Risk/confusion:** Mermaid loads from CDN; offline shows source text (acceptable degradation).
- **Change (v2.3):** Per-mission Board tab removed — **the Plan is the board at mission scale**. The mission Work surface always renders the Plan (nodes carry live status, bound sessions, evidence); the fleet keeps the All Agents Board. `state.rightTab` is replaced by `state.inspectorFocus ("node" | "session")`, which only selects the inspector flavor + node-card highlight (deterministic transition table in design doc §5).
  **Why:** At ≤6 sessions, status-lane grouping added nothing over live node status and created a "which view do I watch?" fork mid-flight. Also resolves former open question "plan as list vs board mid-flight."
  **User-visible result:** Work view header reads "Mission Plan · Live — nodes carry their session, status, and evidence"; selecting a session anywhere opens the session-flavored inspector over the Plan.
  **Risk/confusion:** None expected; users who want lanes have the fleet board.

- **Change (v2.3):** The **decision stat** — `mission.userDecisions` records a fixed label for every user interaction that resolved a checkpoint or steered the mission (full counting table in design doc §4; auto-approved steps and autonomous re-steers never count). Completion posts a "**Needed you N×** — plan approval · step check-in ×4 · confirmed the merge" stat card; the status strip gains a green `needed you N×` pill.
  **Why:** Decisions-per-mission *is* the mode's value metric; it was narrated but never shown. One line converts "trust me, it's less work" into a number — and it's honest under Step (7× vs Auto's 2× is the pace comparison working).
  **User-visible result:** Stat card as the mission's closing beat; pill on completed missions.
  **Risk/confusion:** A large N on Step could read as an indictment — it's the intended argument for Auto-with-gates.

## 4. Selection Rules

| User action | Expected result | Source of truth |
|---|---|---|
| Select Mission (rail row) | `coordView=mission`, split renders (thread+plan), inspector closed, checkpoint choice cleared, gate disarmed; `selectedSession` = first delegated session of that mission; `selectedNode` = first plan node | `state.selectedMission`; mission object for derived fields |
| Select All Agents Board | `coordView=all`, inspector closed; lanes rebuilt from `boardSessions()` (non-example, non-sub-agent) | Session list (status field groups lanes) |
| Select Decisions | `coordView=decisions`, inspector closed; cards rebuilt from `decisions()` (derived, sorted by urgency) | Derived from mission `approval`/`checkpoint` + session `status`/`runState`; never stored |
| Select decision card | Route to owning mission (split renders). With `sessionId`: if the session binds to a plan node → node selected, its card expands (no inspector); else session inspector overlays the plan pane. Without: nothing extra — the checkpoint is inline in the thread (left pane) | Decision's `missionId`/`sessionId`; `planNodeForSession` resolves the node |
| Select delegated session card (board/thread card) | `selectedSession=id`, session inspector overlays the plan pane (fleet: trailing inspector as before) | `state.selectedSession`; session object |
| Open Agent Session (inspector/card button) | Surface switches to Agent (⌘1), `agentSelection=session`, `agentSession=id` | `state.agentSession`; session transcript is the content |
| Back to Mission ("Open Mission →" in attribution bar) | Surface → director, mission selected, `missionView=talk`, inspector closed; the delegated card is scroll-to-centered and its plan node selected if bound | Session's `mission` id; `planNodeForSession` for node; `state.scrollToSession` (transient, mock helper) |

| Open plan node from elsewhere (`open-plan-node`) | Same as above plus surface/mission/work-view forcing | Same |

Determinism rule: node→session binding is stored one-way on the node (`node.session`); session→node is resolved by lookup (`planNodeForSession` walks `mission.nodes`). Owner attribution on a session is `s.mission` (delegated) or `s.parentSession` (sub-agent) — never title parsing (matches the Swift lineage-walk invariant).

## 7. Interaction Contracts

### Interaction: Start New Mission
- Trigger: `+` in Director rail → type directive → Enter (or send).
- Preconditions: non-empty directive.
- State changes: new mission (Draft, Awaiting approval, r1, `checkpoint=approval`) with **shape inferred from the directive** (`mission.shape`), pace/checkpoint toggles from the composer (policy pre-loads them; composer values win), policy snapshot (`policyName/policyDone/policyPins`), empty `userDecisions`; grounding messages appended: user directive, skill card (skills = inferred-from-directive ∪ policy pins; empty → "the repo's conventions"), MCP `get_issue` card **only if** directive matches `#NNN`, one grouped **Grounding** card (Skills · Issue · Guidance rows), director grounding line **ending with the shape announcement** (inference input = directive + standing guidance) plus, when a non-empty policy Done re-targets the close against the directive's words or the shape's default, an explicit **close-conflict note** ("…the policy wins unless you say otherwise before I draft" — the override is announced, never silent) (fixed format: "This reads as a {shape} — {reason}. … (Not the shape you meant? Say so below before I draft.)"). If `cpPlanApproval=false`: skip approval — plan drafted immediately, status Running, `checkpoint=step` (Auto then runs; Step pauses at step 1); the grounding line states the shape and that it's running.
- UI changes: mission selected, Talk view, inspector closed.
- Should call runtime/MCP/tool? yes — skill read + issue fetch (MCP) in the real app; mock fakes both.
- Failure state: empty input no-ops. No issue referenced → no faked issue; grounding line says so.
- Notes: `#298` returns the canonical issue body; other refs echo the directive (mock-only detail).

### Interaction: Start New Session
- Trigger: Agent Mode start screen → type → Enter.
- Preconditions: non-empty input.
- State changes: new standalone session (Working/running, `mission:null`), current checkout worktree.
- UI changes: Agent surface, session opened.
- Should call runtime/MCP/tool? yes — real session spawn.
- Failure state: empty input no-ops.
- Notes: appears on All Agents Board (standalone counts); never on any mission board.

### Interaction: Approve Mission Plan
- Trigger: approval checkpoint → Proceed → primary button (label = **"Draft the plan"** when no plan exists yet, **"Approve plan"** after a revision).
- Preconditions: `checkpoint=approval`.
- State changes: `approval=Approved`, `status=Running`, checkpoint marker message; if no plan yet → `draftPlanForMission` (the inferred shape's `SHAPE_LIBRARY` blueprint instantiated; `$mission` substituted; role→flag mapping applied; **non-empty `policyDone` re-targets the land node's terminal** — title/desc/terminal replaced with "(Close set by your policy's Definition of Done.)" suffix, reversibility gate follows the new terminal; ledger initialized; overview reads `**Shape:** … (inferred from your directive)` + probeContext note) + director summary; `checkpoint=step`. Records `plan approval` in `userDecisions` (including the auto-proceed when the user toggles plan-approval off while waiting — a waiver is a decision).
- UI changes: plan populates Plan tab; status strip counts appear; Auto begins step 1 after a beat.
- Should call runtime/MCP/tool? yes — plan drafting is a model call in the real app.
- Failure state: Revise → holds at approval with guidance; Stop → mission Stopped.
- Notes: approving **is** what drafts — no plan exists before this gate.

### Interaction: Gather Evidence (pre-plan probe steering)
- Trigger: at the approval checkpoint (no plan yet), send composer text naming areas.
- Preconditions: `checkpoint=approval` and no workstreams.
- State changes: user message + cyan Probe card appended; `probeContext += text`; records `context steer`; **shape re-inferred over `objective + probeContext.join(" ")`** — if changed, `mission.shape` updates and the Director announces "…that changes the shape: this now reads as a {shape} ({reason})…"; otherwise the standard acknowledgment. On later drafting, overview gains "Also scoped (you pointed me at): …". Re-inference exists **only pre-draft**; after drafting, shape changes are plan revisions.
- UI changes: cards in thread; checkpoint remains.
- Should call runtime/MCP/tool? yes — real read-only probes.
- Failure state: —.
- Notes: replaced a fixed "Gather evidence" button. (`gatherEvidence()`'s three parallel probe sessions are a retained mock path used by the runner; the contract is the free-text steering above.)

### Interaction: Deepen Plan (revise mid-flight)
- Trigger: ✎ Plan input in the **plan pane's footer** (v2.5 — editing lives where the plan lives) → send (`revisePlan`).
- Rules (v2.8, the immutability ladder): completed steps immutable (their evidence stands); running steps take steers via refine/re-steer, never topology edits; only pending topology is editable. Reshape symmetry: under `reshape:auto` the user's revision applies immediately and the run continues; under `reshape:ask` it pauses to the approval checkpoint (the change carries into the shared record). Counted as `plan revision` either way.
- Preconditions: plan exists.
- State changes: "Plan change (from Plan view): …" user message; new Pending node appended to the implementation workstream with Markdown desc + mini Mermaid diagram, flagged `added`; overview flowchart extended with the new node; revise checkpoint marker; `approval=Awaiting approval`, `checkpoint=approval` (plan changes always pause, even on Auto); runner cleared.
- UI changes: purple "✎ New step" card in plan; approval checkpoint in thread.
- Should call runtime/MCP/tool? yes — model drafts the revision.
- Failure state: empty/no-plan no-ops (input disabled pre-plan).
- Notes: revision requires re-approval by design (open question: what may "revise" change without re-approval).

### Interaction: Get Independent Critique (Review + re-review)
- Trigger: Review node runs as a fresh read-only delegated session; at the step checkpoint after it completes, composer feedback = re-review.
- Preconditions: review node completed; paused at step checkpoint, nothing running (`canRefineStep`).
- State changes: `refineLastStep` — node back to Running; **same** session re-engaged (steer message + "Re-reviewing…" transcript line); refine checkpoint marker; no new session/card.
- UI changes: composer placeholder switches to "Feedback to refine '…' — or Continue above to accept…".
- Should call runtime/MCP/tool? yes — steers the live session.
- Failure state: —.
- Notes: first review = fresh eyes (new session, intentional); every re-pass steers.

### Interaction: Step Continue
- Trigger: step checkpoint → Continue.
- Preconditions: clean boundary — nothing Running; checkpoint shows only then (plus Step pace or an armed gate).
- State changes: if the next Pending node is gated (`mutableGate`) on Step and the gate isn't armed → arm it (amber "Before the first mutable write" required checkpoint), record `step check-in`, and stop; otherwise record the decision (`before-edits gate` if this confirm crosses a gate — armed on Step, or direct on Auto — else `step check-in`; the final all-complete confirm is uncounted) and `startNextNode`: **all** ready nodes start together (real parallelism; land never batched with others; a `compareChoice` node instead sets `checkpoint=pickwinner`), batch-aware checkpoint marker posted ("N steps in parallel · …" or "Step i of N · …"); delegation messages + session cards; sub-agents spawned on first implement pass only; land node emits the terminal bash card (`gh pr create` / `gh issue create` / report) and sets landPhase. No ready nodes → mission finishes.
- UI changes: cards animate to Working; Auto shows "» auto-approved" markers instead of waiting.
- Should call runtime/MCP/tool? yes — delegation is real session spawning.
- Failure state: gate on Auto: the runner never auto-crosses a gated node (halts for approval).
- Notes: readiness = explicit `after` deps all Completed, else implicit previous node.

### Interaction: Evidence bar miss (supervised re-steer)
- Trigger: a running node with `barMiss` reaches its would-be completion tick for the first time (mock trigger: scripted on the Scoped Change implement node; real trigger: evidence verification fails the node's "Done when").
- Preconditions: node Running; miss not yet played this mission.
- State changes: ✗ evidence message (`miss:true`) with the partial evidence + the named gap; director line "Doesn't meet the bar yet — <gap>. Steering the same session…"; the bound session gets a steer transcript entry and stays the same session (no spawn); node **stays Running**; nothing enters the ledger. Next tick completes normally: ✓ card, "meets the bar — advancing," ledger entry.
- UI changes: amber ✗ card in thread (Turn Float peek shows "✗ Evidence · … — short of the bar"); board card unchanged (same session, still Working).
- Should call runtime/MCP/tool? yes in the real app — this **is** the verification fail transition; the re-steer is a real session turn.
- Failure state: n/a in mock (single scripted miss). Real app: repeated misses need an escalation rule (max re-steers → Blocked/decision).
- Notes: plays once per mission (`_barMissPlayed` persists across re-passes so Request-changes never replays it). On Auto the re-steer is autonomous — the bar is enforced without a user stop.

### Interaction: Step Revise
- Trigger: step checkpoint → Revise.
- Preconditions: any non-merge checkpoint.
- State changes: director "Holding for a plan change…" line; `checkpoint=approval`; runner cleared; gate disarmed.
- UI changes: approval checkpoint replaces step checkpoint; composer directs edits.
- Should call runtime/MCP/tool? no (routing only).
- Failure state: —.

### Interaction: Stop Mission
- Trigger: Stop option on any checkpoint (including merge).
- Preconditions: —.
- State changes: `status=Stopped`, `approval=Stopped`, `checkpoint=null`; at merge, PR stays open ("nothing was merged"); runner cleared.
- UI changes: red Stopped pill; composer status dot flips.
- Should call runtime/MCP/tool? no in mock; real app should stop child runs (open question: cancel vs abandon children).
- Failure state: —.

### Interaction: Open Agent
- Trigger: "Open Session"/"Review changes" button (inspector/board), or Agent sidebar row.
- Preconditions: session exists.
- State changes: surface=agent, session opened.
- UI changes: full Agent Mode transcript; attribution bar iff delegated; sub-agent panel iff children.
- Should call runtime/MCP/tool? no.
- Failure state: —.

### Interaction: Back to Mission
- Trigger: attribution bar "Open Mission →".
- Preconditions: session has an owner mission.
- State changes: surface=director, mission selected, Talk view; delegated card scroll-centered; bound plan node selected.
- UI changes: mission thread with the session card highlighted in view.
- Should call runtime/MCP/tool? no.
- Failure state: —.

- See §4 row. No runtime calls; selection + inspector only.

### Interaction: Select Board Card
- See §4 row. No runtime calls; selection + inspector only.

### Interaction: Propose / Apply Worktree Split (mid-flight escalation)
- Trigger: **mock-only trigger** — "Propose split" button on the Plan (`canProposeSplit`: a plain single-worktree implement node exists, not completed, split not yet applied). Real app: Director surfaces it from discovery evidence.
- Preconditions: mission Running; no split applied.
- State changes: director proposal message; `checkpoint=splitProposal`. **Split:** implement node deleted and replaced in-place by two `partWorktree` implement nodes (both `after` discovery) + an Integrate node (`after` both); any superseded bound session marked Done ("Superseded…"); completed downstream review/land reset to Pending; overview gains "Revised mid-flight" note; revise marker + resume. **Keep:** director "keeping it as one worktree" line; resume.
- UI changes: plan rewrites; parallel band renders; split suggestion strip disappears.
- Should call runtime/MCP/tool? yes on apply (spawn parallel sessions when they start).
- Failure state: —.

### Interaction: Merge checkpoint (Merge / Request changes / paste PR feedback)
- Trigger: after Land opens the PR, `checkpoint=merge` (always stops — even fully-autonomous runs).
- Preconditions: PR opened (`prOpened`).
- State changes: **Merge** → `gh pr merge --squash --delete-branch` bash card; land node Completed + ledger entry; dwell on output (~2.6s, mock timing) then wrap-up + mission Completed. **Request changes** → implement→land nodes reset to Pending but **sessions kept** (steer, same PR — same rule on the generic Request-changes path); status Running. **Composer text** → same as Request changes with the feedback quoted; next land pass runs `git push` to PR #142 instead of `pr create`. **Stop** → Stopped, PR left open.
- UI changes: amber "Merge checkpoint — Always stops"; Decisions gains "Review & merge the PR".
- Should call runtime/MCP/tool? yes — gh CLI operations.
- Failure state: not modeled (mock); real app needs push/merge failure states.
- Notes: issue/report terminals skip this gate entirely (reversible closes don't confirm).

### Interaction: Pick the winner (bake-off)
- Trigger: `compareChoice` node becomes ready → `checkpoint=pickwinner` (stops even on Auto).
- Preconditions: both alt nodes Completed with evidence.
- State changes: side-by-side evidence rows (tests + benchmark); choice A/B → compare node Completed, ledger entry ("Picked Alt X…; the other worktree is discarded"), director line; resume to land.
- Should call runtime/MCP/tool? yes on merge of winner.
- Failure state: —.

### Interaction: Autonomy policy (the Autonomy menu — v2.4)
- Trigger: `Autonomy · asks N` pill in composer (draft or live mission).
- Preconditions: —.
- State changes: each class row toggles `ask ↔ auto` on `draftAutonomy` (draft) or `mission.autonomy` (live). Live rules: `writes` re-mirrors every `mutableBoundary` node's `gate`; `plan → auto` while waiting at approval = delegating the decision — the Director proceeds immediately (counts as `plan approval`). The Step-advance row toggles pace (the composer seg is its shortcut). Bar recovery and Irreversible are locked.
- UI changes: pill count = ask classes + (Step ? 1 : 0) + 1; ✋/⚙ row states; gates appear/disappear on node cards.

### Interaction: Child question (`childAsk` class — v2.4)

- Preconditions: node Running with `childAsk` payload, not yet played.
- State changes: amber `childq` card always posts. **ask:** `checkpoint="childAsk"`, session `runState=waitingForQuestion`; the checkpoint shows the question + the Director's suggested answer; primary button sends it, composer text sends a custom answer (`answerChildQuestion`); either records `answered a child question`, posts the Director's answer, flips the session back to running, resumes. **auto:** the Director answers from mission context — ⚙ card quoting the answer + why it was answerable (`answered the implementer` logged) — and the run never halts.
- UI changes: checkpoint variant / ⚙ card; composer placeholder switches to "Type your own answer…".
- Should call runtime/MCP/tool? yes — the interception decision is a model call (Prompt Design §7): answerable-from-context vs escalate-with-suggested-answer; escalations always carry a one-tap answer.
- Failure state: low confidence on auto → escalate anyway (auto = allowed, not required).

Non-empty Done → `terminalFromDone` drives the drafted-close + reversibility hint; empty Done → "each mission closes the way its drafted shape ends".
- UI changes: live drafted-close hint only when Done is non-empty; no brief field; no plan preview (deleted machinery listed in design doc §3.4/§8 — must not be resurrected).
- Should call runtime/MCP/tool? no — a policy is pure configuration.
- Failure state: unnamed → "Untitled policy".

### Interaction: Follow-up mission (v2.7)
- Trigger: any composer message into a mission whose `status ∈ {Completed, Stopped}` (the placeholder announces: "…your message starts a FOLLOW-UP mission that inherits its receipt as context").
- Preconditions: none beyond status — the finished mission is immutable; there is no reopen path.
- State changes: `startNewMission(text, prevId)` creates a NEW mission: own shape inference on the new directive (predecessor context deliberately excluded from inference — the directive speaks for itself), `followsFrom = prevId`, policy + pace + autonomy inherited from the predecessor (earned trust carries forward), subtitle "Follow-up of …".
- UI changes: Grounding card gains the `◇ Follows` row (predecessor title + receipt digest: evidence rows · needed/decided totals · close line); delegation prompts append the inherited-receipt instruction; the predecessor's thread/stat/receipt are byte-identical before and after (harness-verified).
- Should call runtime/MCP/tool? In Swift, the inherited context = the predecessor's receipt projection injected into the §2 drafting and §3 delegation prompts (a scoped context package, not a transcript).
- Failure state: —.

### Interaction: Mission Receipt (v2.6)
- Trigger: "Mission receipt →" on the completion stat card, or the **Plan | Receipt** seg in the plan pane header (v2.7 — present only when Completed).
- Preconditions: mission `status === "Completed"` — a live plan is never covered by the receipt.
- State changes: `receiptOpen` flips (opening also un-collapses the plan pane); reset on mission switch / new mission.
- UI changes: the plan pane renders the receipt (Mission / Contract / Decisions incl. ⚙ reasons / Evidence / Close) with **Copy Markdown** (writes `missionReceiptMarkdown(m)` to the clipboard) and **‹ Plan**.
- Should call runtime/MCP/tool? no — the receipt is a pure projection of mission-owned state; an optional model-composed "learnings / next step" line is a later addition, not v1.
- Notes: this is the source-neutral accountable record; the PR body (Prompt Design §10) and the receipt are the same projection aimed at different readers.

### Interaction: Decision accounting (cross-cutting — two-sided in v2.4)
- Trigger: every user interaction that resolves a checkpoint or steers the mission (user side; table in design doc v2.3 §4, + `answered a child question`), and every decision the Director takes on an auto class (auto side).
- State changes: fixed label appended to `mission.userDecisions` (user) or `mission.autoDecisions` + a `role:"auto"` ⚙ card with a one-line reason (Director; advance class = "» auto-approved" markers + `_autoAdvances` counter, no cards). Bar recovery always logs `re-steered a short bar (same session)`. Never counted as user decisions: auto-approved steps, autonomous re-steers, probe/loop/psteer beats, option selection without confirm, the final all-complete confirm.
- UI changes: on completion `finishMission` posts the two-line `role:"stat"` card ("Needed you N× — …" / "Decided itself M× — …", parts grouped ×k, auto side folds in `advanced past N steps on evidence`); completed missions' strip shows the green `needed you N×` + grey `decided itself M×` pills; Turn Float peek renders both totals.
- Should call runtime/MCP/tool? no (Swift: persist as typed events — label + checkpoint id + timestamp).
- Notes: honest under Step by design (7× vs Auto's 2× is the pace comparison working); a revision counts twice (revision + re-approval) because it is two interactions.

## 8. Source Of Truth Rules

| Information | Source of truth | Projected into |
|---|---|---|
| Mission status | Mission object (`status`) | Rail row pill/plate, status strip, composer status dot, Decisions derivation |
| Plan approval | Mission (`approval` + `cpPlanApproval` policy) | Approval checkpoint, Decisions ("Approve the Mission plan"), status strip |
| Plan nodes | Mission plan (`workstreams` + `nodes`) — the plan is Mission-owned output, never template-owned | Plan tab cards/bands, Plan Inspector, status strip counts, checkpoint step labels, board via node→session binding |
| Session status | Session object (`status` lane + `runState`) — the run/session state control plane | Board lanes, rail/sidebar plates, inspector, delegated cards in thread, Decisions (blocked / needs-you / review) |
| Worktree identity | Session (`worktree`, `branch`); plan node carries only the *decision label* (`partWorktree`/`altWorktree`/derived `worktreeFor`) | Node ⌥ wt/… line, inspector Worktree rows, card plate dot color, Agent header |
| Shape decision | `mission.shape` + the grounding announcement message (shape + reason + override invite = the routing record); re-inference pre-draft only | Shape announcement in thread, approval checkpoint sub-line, plan overview `**Shape:** …` line |
| Routing decision (workflow, policy, worktree choice, skills) | Derived per node from what the step does (role/flags → `policyRationale`, `worktreeFor`; skills from directive inference ∪ policy pins) with user override; never a typed enum | Node "why" line, inspector Why/Worktree |
| Decision count (two-sided) | `mission.userDecisions` + `mission.autoDecisions` (+ `_autoAdvances`), append-only fixed labels (design docs v2.3 §4 / v2.4 §2-3) | Two-line completion stat card, `needed you N×` + `decided itself M×` strip pills, ⚙ cards, Turn Float peek |
| Evidence | Mission ledger (`evidence[]`, append-only) + per-node `done`/`evidence` | Ledger panel, ✓ evidence cards in thread, node cards (Done when / ✓), inspector, bake-off comparison rows, PR description (stated, not modeled) |

## 9. Visual Rules

- Layout (v2.5): rail 248px (220px narrow) + the persistent mission split — thread `minmax(380px, 44%)` (46% narrow) + plan pane `1fr`; the session inspector overlays the plan pane (`min(384px, 86%)`, close-handle) instead of squeezing a third column; the fleet board keeps its trailing inspector.
- Density (v2.2): node cards are **compact by default** — title/status, workflow chip (colored by kind: Review purple, Explore blue, Orchestrate green, Investigate/Validation cyan, Deep Plan amber, Optimize red), gate chip (amber), ✎ New step chip, ⌥ worktree label (no note), loop strip, Done-when, ✓ evidence when complete. **Selection expands the card:** "why" rationale, worktree note, policy (blue)/deps/sub-tasks/binding chips (Bound session green / Steers prior session blue / Director-direct / Plan intent only), Markdown desc + detail. Workstream headers show the strategy chip only. Redundant chips removed earlier: no "tests" step chip (folded), no Review board lane (Review is a workflow, not a status), no outcome-type chip (closing is drafted + reversibility-flagged).
- Panels collapse: workstreams auto-collapse when all nodes complete (user toggle overrides, sticky per id); Examples collapsed by default in both rails; checkpoint menu is a popover; inspector is summoned, not persistent.
- Max heights / scrolling: bash output capped 168px (scrolls); Turn Float peek 176px → raised `min(64%, 520px)` with scrim; composer textarea 40–110px; thread/plan/board/inspector each scroll independently; thread and Agent transcript stick to bottom, other scrollers preserve position across renders.
- Mission vs Session look: Mission surfaces carry the status strip + plan/ledger vocabulary and a mission plate glyph; sessions are cards/rows with worktree-colored plate dots and workflow chips. The Agent surface stays visually today's product — the attribution bar is the only Mission element.
- Delegated vs standalone: delegated = attribution bar + "Mission: …" sidebar subtitle + owner chip on the fleet board; standalone = "Standalone session" subtitle, no bar, no owner chip. Sub-agents nest under their parent in the sidebar and in a "Agent-owned sub-agents" panel inside the parent transcript; they never appear as top-level board cards.
- Motion/dwell: merge/issue/report CLI output dwells (~2.6s) before the wrap-up lines; parallel bands intentionally dwell (~8 beats) to show per-thread re-steering; loop iterations tick faster (1.7s) than step completion (3.5s). All timings mock-only, but "terminal output is readable before the wrap-up" is a product rule.
